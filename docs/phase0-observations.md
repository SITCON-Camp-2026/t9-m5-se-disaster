# Phase 0 Observations

## 我們做出了什麼？

- 整理工作台：可以查看 12 筆原始資訊，為每筆建立、編輯、刪除整理草稿
- 預設 6 筆整理草稿（M-001, M-002, M-009, M-010, M-011, M-012），展示不同品質等級的整理範例
- 左側列表標示已建立草稿的資訊（✓ 符號）
- 可編輯的判斷欄位：候選類型、信心程度、判斷依據、卡住的地方、下一步建議、人工審視備註
- 分別在「原始資訊」頁面直接顯示資訊列表，標示「這還不是整理後資料」

## 哪些資料讓 agent 很容易整理？

- **M-010（溪畔活動中心現場更新）**：時間、地點、具體數字、狀態清晰，無歧義。可直接變成候選地點狀態更新。
- **M-009（光復車站集合點）**：現場志工實時報告，有時間戳、限制條件明確，品質最高。
- **M-003（老街口不缺鏟子）**：明確的物資狀態更新和反饋。

## 哪些資料讓 agent 亂補或誤判？

- **M-001（光復車站清泥）**：原文只說「老雜貨店後面」，系統很容易補假設的完整地址。但實際上這個地址無法精確定位，不能直接派人。
- **M-004 vs M-002 的衝突**：不同時間、不同來源的「雨鞋資訊」互相矛盾。系統可能會誤認為是重複資訊，其實是過時資訊。
- **M-007（工班支援）**：社群貼文說「某工班可以支援」，但留言說「今天沒空」。系統容易只取前半部分當成承諾。
- **M-008（A 區先不要派人）**：現場反報沒有說明理由（人太多？道路危險？任務完成？）系統無法判斷。

## 哪些資訊不能直接變成任務？

- **M-001**：地址不精確，無法派人
- **M-002 / M-004**：時間過時，無法確認現況
- **M-005**：沒有日期和來源驗證，「中午前」對誰有效？
- **M-006**：學校側門資訊相衝，需要現地確認
- **M-007**：工班可用性不確定
- **M-008**：不知道為什麼不能派人
- **M-011**：當事人（長者）未直接同意，不能直接派人上門
- **M-012**：當事人不在場，無法確認

## 哪些資訊的操作者不是當事人？

- **M-011（長者家具搬運）**：志工代轉述，長者本人沒有直接確認
- **M-012（外地家屬代轉藥品需求）**：親友本人不在現場，家屬無法確認親友位置和意願
- **M-006（學校側門集合點）**：有人說可以、有人說淹水，不知道誰有現地確認權

## 我們剛剛偷偷假設了什麼？

- 假設「原文明確說『需要』」就代表確認無誤。其實需要詢問「誰說的」「怎麼確認」「這是當事人說的嗎」。
- 假設時間戳近就代表資訊準確。但 M-002 說「不知道下午還有沒有」，就已經是過時資訊。
- 假設「志工回報」就比「社群貼文」可信。其實兩者都需要查核來源和當事人意願。
- 假設有具體地名就能直接派人。但「老雜貨店後面」和「大進路口往溪邊方向第二排住家」都不能精確定位。

## 如果要讓這個工具變可靠，下一步需要什麼？

1. **當事人確認流程**：標示操作者身份，區分「當事人確認」「志工轉述」「第三手資訊」
2. **時間有效性**：標示資訊蒐集時間、預期失效時間、是否需要重新查核
3. **地址標準化**：建立地址確認機制，不要直接相信模糊地址
4. **來源驗證**：標示來源可信度等級，官方公告是否已同步
5. **動態更新**：現場情況變化快，需要標示「上次確認時間」和「需要重新查核」的觸發條件
6. **任務前提**：清楚標示每筆資訊能否直接變成任務，缺少什麼才能變成任務

## 我們需要更清楚資料格式的地方

- `sourceType` 只代表資訊來自哪個管道，不代表確認狀態。需要加上「操作者身份」（當事人 / 志工 / 第三方）
- `verificationStatus` 應該分兩個維度：「資訊查核狀態」（verified / needs_review / unverified）和「當事人確認」（親自確認 / 代轉述 / 未確認）
- 需要「資訊時效性」欄位：「實時」「預期有效到 xx 時間」「需要重新查核」

## 我們需要人工確認的地方

- M-002 / M-004 現在還有沒有雨鞋
- M-005 官方公告日期和有效範圍
- M-006 學校側門現在能不能停留
